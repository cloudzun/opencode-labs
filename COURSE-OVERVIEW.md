# OpenCode Labs 课程全景

## 课程定位

本课程是一套面向开发者的 **AI 协同编程**实战手册，帮助学员系统掌握以 OpenCode 为代表的终端 AI 编程工具，在真实项目中建立从需求到交付的完整 AI 协同工作流。

> **AI 协同编程**（AI-Collaborative Programming）：人类保持架构决策权与质量判断，AI 负责代码执行与方案生成，两者在每个开发环节持续交互、相互增强。这不是"放手让 AI 写"，而是人与 AI 各司其职、协同完成软件工程的全过程。

---

## 设计原则

### 原则一：Prompt 驱动，而非手写代码

所有代码操作均通过自然语言 Prompt 驱动 OpenCode 完成。学员的核心技能是**写好 Prompt**，而不是记忆代码语法。

### 原则二：验证优先

每个实验手册在发布前均经过实际运行验证——从干净环境克隆项目，按手册 Prompt 逐步执行，编译通过、测试通过才算完成。

### 原则三：循序渐进，螺旋上升

课程从"理解代码"出发，经过"开发功能"、"质量保障"、"高级模式"，最终到达"规格驱动开发"和"综合实战"。每个阶段都在前一阶段的基础上引入新的复杂度。

```
LAB-00/01  环境与工具
    ↓
LAB-02~05  基础开发工作流（分析、开发、测试、重构）
    ↓
LAB-06     快速原型（Vibe Coding 模式）
    ↓
LAB-07~08  多代理协同（并行 Agent、子代理分析）
    ↓
LAB-09~11  自定义 Agent（代码审查、性能、安全）
    ↓
LAB-12~13  规格驱动开发（Greenfield + Brownfield）
    ↓
LAB-14     综合实战（完整软件工程周期）
```

### 原则四：决策优于操作

手册不给"正确答案"，而是给**思考框架**。学员需要判断何时重构、如何权衡性能与安全、什么时候引入 SDD——这些工程决策能力才是真正的学习目标。

### 原则五：真实项目，真实问题

所有实验均基于真实的开源项目（Microsoft Learning 系列），遇到的问题（SQLite 迁移、编译错误、数据库迁移失败）都是真实发生并解决的，不是人为构造的练习题。

## 实验全景

### 模块一：入门准备（LAB-00 ~ LAB-01）

| 实验 | 核心内容 | 学习收获 |
|------|---------|---------|
| LAB-00 | OpenCode 安装（Windows/Linux/macOS）+ 百炼 Coding Plan 配置 + Prompt 驱动安装开发环境 | 从零到可用的完整环境 |
| LAB-01 | TUI 界面、三种模式（explore/plan/build）、@ 文件引用、/init 命令、自定义 Agent 基础 | 掌握 OpenCode 全部核心操作 |

**模块特点**：不假设任何 OpenCode 经验，从安装开始，用 Prompt 完成环境配置，体现"工具本身就是工具"的理念。

---

### 模块二：基础开发工作流（LAB-02 ~ LAB-05）

| 实验 | 核心内容 | 学习收获 |
|------|---------|---------|
| LAB-02 | 用 explore 模式分析 Python 代码库，生成架构文档、函数说明、README | 建立"先理解再动手"的习惯 |
| LAB-03 | 用 build 模式为现有项目添加新功能（患者记录管理系统） | 掌握功能迭代的 Prompt 模式 |
| LAB-04 | 用 OpenCode 生成单元测试，覆盖正常路径、边界条件、异常处理 | 理解测试驱动的 AI 协同模式 |
| LAB-05 | 代码重构：消除重复、提取函数、改善命名、提升可读性 | 掌握渐进式重构的 Prompt 策略 |

**模块特点**：每个实验对应一个独立的开发工作流节点，学员可以单独学习任意一个，也可以按顺序体验完整工作流。

---

### 模块三：快速原型（LAB-06）

| 实验 | 核心内容 | 学习收获 |
|------|---------|---------|
| LAB-06 | 从零 Vibe Coding 一个电商原型：产品列表、购物车、结账流程 | 体验"想到哪做到哪"的快速原型模式 |

**模块特点**：这是课程中唯一不强调代码质量的实验，目的是让学员感受 AI 协同编程的速度上限，以及快速原型的代价（技术债）。

---

### 模块四：多代理协同（LAB-07 ~ LAB-08）

| 实验 | 核心内容 | 学习收获 |
|------|---------|---------|
| LAB-07 | 并行启动多个 OpenCode 实例，同时重构多个独立模块的重复代码 | 掌握并行 Agent 的任务拆分策略 |
| LAB-08 | 子代理深度分析大型函数，主代理按分析报告执行重构 | 理解"分析与执行分离"的协同模式 |

**模块特点**：引入多代理概念，学员开始思考"如何把任务分配给多个 AI"，而不只是"如何写一个好 Prompt"。

---

### 模块五：自定义 Agent（LAB-09 ~ LAB-11）

| 实验 | 核心内容 | 学习收获 |
|------|---------|---------|
| LAB-09 | 创建 @code-reviewer Agent，审查并简化复杂条件语句（295 行 → 30 行） | 掌握自定义 Agent 的设计与调用 |
| LAB-10 | 创建 @performance-auditor Agent，识别并修复性能瓶颈 | 建立性能优化的系统化思路 |
| LAB-11 | 创建 @security-auditor Agent，三阶段并行安全修复流水线 | 掌握安全加固的 Agent 协同模式 |

**模块特点**：学员从"使用 AI"升级到"定制 AI"，为特定场景创建专属 Agent，形成可复用的团队工具。

---

### 模块六：规格驱动开发（LAB-12 ~ LAB-13）

| 实验 | 核心内容 | 学习收获 |
|------|---------|---------|
| LAB-12 | SDD Greenfield：从零创建 RSS Feed Reader，@spec-writer Agent 生成四层文档（constitution → spec → plan → tasks） | 掌握文档驱动开发的完整流程 |
| LAB-13 | SDD Brownfield：为现有 ContosoDashboard 添加文档管理功能，分析现有架构约束后设计集成方案 | 理解在现有项目中引入 SDD 的方法 |

**模块特点**：课程的技术高峰。学员学会在动手之前先建立系统性的规格文档，让 AI 在有明确约束的框架内执行，大幅降低"AI 乱改代码"的风险。

---

### 模块七：综合实战（LAB-14）

| 实验 | 核心内容 | 学习收获 |
|------|---------|---------|
| LAB-14 | 五阶段完整软件工程周期：Vibe Coding 原型 → 功能迭代 → 性能+安全加固 → SDD 重构 → 复盘沉淀 | 建立完整的工程决策直觉 |

**模块特点**：唯一没有标准答案的实验。验收标准是工程决策的合理性，而不是代码是否跑通。引入 Shopify CEO 的上下文文档框架，学员体验"写好上下文 = 写好 Prompt"的核心认知。

## OpenCode 与 GitHub Copilot（VS Code）的使用场景差异

两者都是 AI 辅助编程工具，但面向不同的使用场景，没有优劣之分。

| 维度 | OpenCode（终端 TUI）| GitHub Copilot（VS Code 插件）|
|------|---------------------|-------------------------------|
| **交互方式** | 终端对话，全键盘操作 | IDE 内嵌，实时代码补全 |
| **适用场景** | 大范围重构、多文件协同、自动化任务 | 行级补全、函数生成、即时建议 |
| **上下文管理** | 手动用 @ 引用文件，精确控制 | 自动感知当前文件和光标位置 |
| **Agent 能力** | 支持自定义 Agent、并行多实例 | 内置 Agent 模式（GitHub Copilot Chat）|
| **Provider 灵活性** | 支持 75+ 提供商，可接入百炼等国内服务 | 主要使用 GitHub/Azure 托管模型 |
| **工作流集成** | 适合 CI/CD、脚本化、无头环境 | 适合 GUI 开发、调试、可视化操作 |
| **学习曲线** | 需要适应终端操作方式 | 对 VS Code 用户几乎零学习成本 |

**一句话总结**：GitHub Copilot 是"在编辑器里随时获得建议"，OpenCode 是"把一个任务完整交给 AI 执行"。两者互补，很多开发者同时使用。

## 关于本课程的创作方式

本课程手册由**三方协同创作**完成：

- **人类（课程设计者）**：负责课程架构设计、实验目标定义、质量把关、最终审核
- **OpenClaw**：负责任务编排、子代理调度、进度管理、发布流程
- **OpenCode**：负责手册内容生成、代码验证、问题修复、文档优化

这种创作方式本身就是 **AI 协同编程**理念的体现——人类定义"做什么"和"为什么"，AI 负责"怎么做"的执行细节。整个课程的生产过程，就是课程所教内容的最佳示范。
