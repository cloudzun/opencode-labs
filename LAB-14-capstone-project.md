# LAB-14：Capstone Project - AI 驱动完整软件工程周期

---

## 1. 实验概述

### 综合实战：用 AI 驱动完整软件工程周期

**学习目标**
- 掌握上下文文档的编写方法，理解「上下文优于提示词」的 AI 协作原则
- 体验产品从 0 到 1 的完整演进周期，建立工程直觉
- 综合运用 LAB-02 至 LAB-13 的核心技能，形成技能网络
- 建立工程决策的判断力，理解技术债、架构演进的权衡

**预计时间**：90-120 分钟（建议分两次完成：Phase 1-3 为第一次，Phase 4-5 为第二次）

**前置条件**：已完成 LAB-02 至 LAB-13，熟悉各实验的核心技能

**核心理念**：
> 这是 OpenCode Labs 中**唯一没有标准答案**的实验。验收标准不是代码是否跑通，而是工程决策的合理性。你模拟的是一个真实产品从原型到成熟的完整生命周期，每个阶段的决策都会影响后续演进。

## 2. 上下文文档框架

### Shopify CEO 的上下文文档习惯

在每次请求 AI 协助之前，先写一份**上下文文档**，而不是提示词。上下文文档不是命令，而是让 AI 理解你所处的情境。

### 五个维度

| 维度 | 核心问题 | 示例 |
|------|---------|------|
| ① 项目背景 | 当前状态是什么？ | 「这是一个 Flask 博客系统，已有用户认证和文章 CRUD 功能」 |
| ② 之前尝试过什么 | 上一阶段做了什么？遇到了什么问题？ | 「尝试过添加评论功能，但数据库设计有缺陷，导致查询性能差」 |
| ③ 成功与失败的定义 | 本阶段结束时，什么算成功？什么算失败？ | 「成功：API 响应时间<200ms；失败：破坏现有用户认证逻辑」 |
| ④ 谁会受影响 | 这次改动影响哪些功能/用户/模块？ | 「影响：/api/posts 接口、前端文章列表页、移动端 App」 |
| ⑤ 约束条件 | 技术约束、时间约束、不能破坏什么？ | 「约束：不能更换数据库、必须在 2 小时内完成、不能破坏向后兼容」 |

### 上下文文档模板

每次进入新阶段前，复制以下模板并填写：

```markdown
# 上下文文档 - [阶段名称]

## 项目背景
[当前系统状态：技术栈、已有功能、架构特点]

## 之前尝试过什么
[上一阶段的成果、遇到的问题、遗留的技术债]

## 成功与失败的定义
- 成功标准：[可量化的目标]
- 失败红线：[绝对不能破坏的东西]

## 谁会受影响
- 受影响的功能模块：[列表]
- 受影响的用户群体：[列表]
- 需要回归测试的内容：[列表]

## 约束条件
- 技术约束：[不能更换的技术栈、必须兼容的接口]
- 时间约束：[预计完成时间]
- 质量约束：[性能指标、安全要求]
```

**关键认知**：写好上下文文档的 AI 输出质量，远高于随手写的 Prompt。上下文文档不只是思考工具，它直接成为给 OpenCode 的 Prompt 上下文。

## 3. 项目背景：个人任务管理 + 团队协作系统（Task Board）

### 项目描述

一个支持个人任务管理和团队协作的看板系统。用户可以创建任务、管理任务状态、与团队成员协作。

### 为什么选这个项目

| 条件 | 说明 |
|------|------|
| 足够简单 | Vibe Coding 阶段 30 分钟能出原型，学员不会被复杂度劝退 |
| 足够复杂 | 后期能暴露性能问题（大量任务查询）、安全问题（权限控制）、架构问题（状态同步） |
| 贴近真实 | 学员有直觉判断好坏，能理解决策的后果 |

### 功能范围

**核心功能（MVP）**
- 任务增删改查（Create/Read/Update/Delete）
- 任务状态流转（Todo → InProgress → Done）
- 用户认证（注册/登录/登出）

**扩展功能（迭代阶段）**
- 团队协作（多用户共享看板）
- 任务评论（支持@提及）
- 文件附件（上传图片/文档）
- 通知系统（邮件/站内信）

**高级功能（优化阶段）**
- 看板视图（拖拽式 UI）
- 数据统计（完成率、耗时分析）
- 性能监控（API 响应时间、慢查询日志）

## 4. 五个演进阶段

### Phase 1: Vibe Coding 快速原型（20-30 分钟）

**背景**

用最快速度让产品跑起来。不考虑架构设计，不考虑代码质量，不考虑扩展性。唯一目标：**让核心功能可交互**。

**上下文文档填写指引**

填写上下文文档时，重点思考：
- 「什么是这个阶段的最小可用产品？」
- 「哪些功能可以砍掉，哪些功能必须有？」
- 「我选择的技术栈，5 分钟后会不会后悔？」

> **工程师的自问**：「现在能跑就行，但我知道这些地方将来会出问题：___」

**验收标准**

- [ ] 应用能成功启动，无启动错误
- [ ] 用户可以注册/登录
- [ ] 用户可以创建任务
- [ ] 用户可以查看任务列表
- [ ] 用户可以修改任务状态
- [ ] 用户可以删除任务

**提示**：参考 LAB-06 Vibe Coding 模式，不要过度思考，先让代码跑起来。如果你发现自己思考「这个架构好不好」，停下来，继续写代码。

**典型陷阱**

| 陷阱 | 表现 | 正确做法 |
|------|------|---------|
| 过早优化 | Phase 1 就开始考虑数据库连接池 | 直接用最简单的 SQLite，Phase 3 再优化 |
| 过度设计 | 一开始就设计微服务架构 | 单体应用，所有代码在一个 repo |
| 完美主义 | 纠结命名、格式、注释 | 能跑就行，Phase 4 会重构 |
| 写测试 | Phase 1 就写单元测试 | Phase 2/3 再补充测试 |

---

### Phase 2: 功能迭代（20-30 分钟）

**背景**

产品有用户了，需要加功能。但代码开始腐烂——重复逻辑、混乱的命名、没有测试。你面临选择：继续加功能，还是停下来重构？

**上下文文档填写指引**

填写上下文文档时，重点思考：
- 「加这个功能会影响哪些现有功能？」
- 「代码中有哪些重复模式可以抽取？」
- 「技术债在哪里？什么时候必须还？」

> **工程师的自问**：「我注意到代码开始出现重复，但现在是否是重构的好时机？」

**验收标准**

- [ ] 至少新增 2 个功能（从扩展功能列表中选择）
- [ ] 能说出至少 3 个技术债的具体位置
- [ ] 至少为 1 个核心函数编写单元测试
- [ ] 使用 @code-reviewer 审查一次代码

**提示**：参考 LAB-03 开发新功能、LAB-07 重复代码整合。记录你发现的技术债，Phase 4 会处理。

**技术债识别清单**

在 Phase 2 结束时，你应该能识别出以下类型的技术债：

| 技术债类型 | 识别信号 | 示例 |
|----------|---------|------|
| 重复代码 | 同样的逻辑出现 3 次以上 | 每个 API 都手动验证用户登录状态 |
| 过长函数 | 函数超过 50 行 | 一个 handle_request 函数处理所有逻辑 |
| 模糊命名 | 变量名是 data、result、temp | `def process(data):` |
| 魔法数字 | 代码中有硬编码的数字 | `if status == 1:`（1 代表什么？） |
| 缺乏验证 | 直接使用用户输入 | `user_id = request.args['id']` |
| 没有错误处理 | 没有任何 try-except | 数据库查询失败直接崩溃 |

**建议**：用 LAB-09 @code-reviewer 扫描代码，它会帮你识别大部分技术债。把识别出的技术债记录在上下文文档中，Phase 4 优先处理。

---

### Phase 3: 性能优化 + 安全加固（20-30 分钟）

**背景**

用户量增长，性能问题开始暴露：慢查询、高延迟、内存泄漏。同时，安全审计发现漏洞：SQL 注入风险、XSS 漏洞、权限绕过可能。资源有限，你必须做出优先级判断。

**上下文文档填写指引**

填写上下文文档时，重点思考：
- 「优化前后如何量化效果？」
- 「哪个安全漏洞的影响范围最大？」
- 「性能瓶颈在哪里？是数据库、网络还是 CPU？」

> **工程师的自问**：「我要先优化性能还是先修安全漏洞？为什么？」

**验收标准**

- [ ] 使用 @performance-auditor 扫描并修复至少 2 个性能问题
- [ ] 使用 @security-auditor 扫描并修复至少 1 个安全问题
- [ ] 有量化数据证明优化效果（如：响应时间从 Xms 降至 Yms）
- [ ] 记录优化决策的理由

**提示**：参考 LAB-10 性能优化、LAB-11 安全加固。不要盲目优化，先测量再行动。

**常见性能问题清单**

| 问题类型 | 识别方法 | 优化策略 |
|---------|---------|---------|
| N+1 查询 | 一个请求触发多次数据库查询 | 使用 JOIN 或批量查询 |
| 缺乏缓存 | 同样的数据重复查询 | 添加 Redis 或内存缓存 |
| 同步阻塞 | 耗时操作阻塞主线程 | 使用异步或后台任务 |
| 大对象传输 | API 返回过多不必要字段 | 分页、字段筛选 |
| 低效算法 | 嵌套循环处理大数据 | 使用哈希表或索引 |

**常见安全漏洞清单**

| 漏洞类型 | 风险等级 | 修复方法 |
|---------|---------|---------|
| SQL 注入 | 高 | 使用参数化查询 |
| XSS 攻击 | 中 | 对用户输入进行转义 |
| CSRF 攻击 | 中 | 添加 CSRF Token |
| 明文密码 | 高 | 使用 bcrypt 或 argon2 加密 |
| 权限绕过 | 高 | 每个 API 都验证用户权限 |

**决策框架**：当性能和安全都需要处理时，先修复「高」风险的安全漏洞，再优化「高频」的性能问题。

---

### Phase 4: SDD 驱动重构（30-40 分钟）

**背景**

架构已经撑不住了。耦合严重、难以测试、新人看不懂代码。是时候用 SDD（Specification-Driven Development）方式进行系统性重构。这不是推倒重来，而是在现有约束下设计更好的架构。

**上下文文档填写指引**

填写上下文文档时，重点思考：
- 「重构的边界在哪里？什么模块不能动？」
- 「现有代码有哪些隐性约束（业务逻辑、用户习惯、第三方集成）？」
- 「新架构要解决的核心问题是什么？」

> **工程师的自问**：「这是 Brownfield SDD，我需要先分析现有代码的约束，再设计新架构」

**验收标准**

- [ ] 生成四层 SDD 文档（constitution/spec/plan/tasks）
- [ ] 完成至少一个模块的重构（如：用户认证模块或任务服务模块）
- [ ] 重构后的模块有清晰的接口边界
- [ ] 重构后有测试覆盖（至少核心路径）

**提示**：参考 LAB-13 Brownfield SDD 模式。SDD 不是银弹，但能帮你建立系统性思考。

**四层 SDD 文档说明**

| 层级 | 名称 | 内容 | 示例 |
|-----|------|------|------|
| L1 | Constitution | 系统根本原则和约束 | 「所有 API 必须验证用户身份」 |
| L2 | Specification | 详细的功能规格说明 | 「任务状态只能是 Todo/InProgress/Done」 |
| L3 | Plan | 实现计划和模块划分 | 「将用户认证抽离为独立模块」 |
| L4 | Tasks | 具体的任务列表 | 「创建 auth.py，移动 login 函数」 |

**重构边界原则**

- **可以改的**：内部实现、函数命名、代码组织、算法优化
- **不能改的**：对外 API 接口、数据库 schema（除非必要）、用户数据结构
- **谨慎改的**：第三方集成、配置文件格式、日志格式

**建议重构顺序**

1. 先重构「最常被修改」的模块（通常是业务逻辑层）
2. 再重构「最混乱」的模块（代码重复最多的地方）
3. 最后重构「最稳定」的模块（工具函数、配置管理）

### Phase 5: 复盘与沉淀（10-15 分钟）

**背景**

项目完成，是时候提炼经验了。这不是对错评判，而是经验提炼——哪些决策效果好，哪些决策有代价，下次如何改进。

**复盘模板**

回答以下问题（每个问题 2-3 句话）：

1. **哪个阶段最顺畅？为什么？**
   - 是因为技术熟悉？上下文清晰？还是 AI 协作高效？

2. **哪个阶段最痛苦？为什么？**
   - 是因为技术债太多？需求不清晰？还是架构约束太紧？

3. **如果重来，Phase 1 的哪些决策会不同？**
   - 技术栈选择？功能范围？代码质量？

4. **上下文文档对 AI 输出质量的影响有多大？**
   - 有写上下文文档 vs 随手写 Prompt，输出质量差异在哪里？

5. **我学到的最重要的一条工程原则是：___**
   - 这句话应该是你下次做项目时会想起的原则

**验收标准**

- [ ] 完成复盘文档（至少回答 5 个问题）
- [ ] 复盘文档中有具体的例子支撑观点
- [ ] 提炼出至少 1 条可迁移的工程原则

**示例回答参考**

> **问题 3 示例**：「如果重来，Phase 1 我会选择 FastAPI 而不是 Flask。因为 Flask 的代码很快就变得混乱，而 FastAPI 的类型提示和自动验证能减少很多低级错误。虽然学习成本高一点，但 Phase 2 加功能时会更快。」

> **问题 5 示例**：「我学到的最重要的一条工程原则是：**先让代码跑起来，再让代码好起来**。Phase 1 过度思考架构是浪费时间，因为问题只有在代码量大到一定程度才会暴露。但前提是：你要有计划地还技术债，不是永远乱写。」

## 5. 工程决策参考

### 困境 1：「功能还没做完，但代码已经很乱了，现在重构还是继续加功能？」

**思考框架：技术债利息计算法**

| 选项 | 一次性成本 | 每日利息 | 适用场景 |
|------|----------|---------|---------|
| 继续加功能 | 低 | 高（每天多花 30 分钟理解混乱代码） | 距离上线<1 周、团队熟悉代码 |
| 立即重构 | 高（2-3 天） | 低（后续开发效率提升） | 项目长期维护、团队有新人 |

**计算公式**：`利息 = 混乱代码导致的额外时间 × 团队人数 × 剩余天数`

当 `利息 > 重构成本` 时，就是重构的时机。

---

### 困境 2：「性能优化和安全修复，先做哪个？」

**思考框架：风险矩阵**

```
风险值 = 影响范围 × 发生概率 × 修复难度

影响范围：1(小) - 5(全系统)
发生概率：1(罕见) - 5(频繁)
修复难度：1(简单) - 5(复杂)
```

**决策规则**：
- 风险值 > 50：立即处理（无论性能还是安全）
- 风险值 20-50：本阶段处理
- 风险值 < 20：可以延后

**经验法则**：安全漏洞的影响范围通常是「全系统」，性能问题通常是「部分场景」。当两者风险值接近时，优先安全。

---

### 困境 3：「要不要用 SDD？项目多大才值得？」

**思考框架：复杂度阈值**

当你开始有以下感受时，就是 SDD 的时机：

| 信号 | 含义 |
|------|------|
| 「我不敢改这段代码，怕破坏其他地方」 | 耦合度过高 |
| 「这个 bug 修完，那里又冒出来一个」 | 缺乏系统性理解 |
| 「新人入职需要 2 周才能看懂代码」 | 文档和架构不清晰 |
| 「每次加功能都要改 5 个文件」 | 模块边界模糊 |

**SDD 不是银弹**：
- 小型脚本（<500 行）：不需要 SDD
- 中型项目（500-5000 行）：可以考虑 SDD
- 大型项目（>5000 行）：强烈推荐 SDD

**最后提醒**：这是你的项目，你的决策，你的经验。手册只给框架，不给答案。好的工程师不是记住答案的人，而是能做出合理决策的人。

## 附录 B：常见问题解答

### Q1：Phase 1 的代码太烂了，我忍不住想重构，可以吗？

**建议**：可以，但记录你花费的时间。Phase 1 的目标是快速验证，如果你花 30 分钟重构 Phase 1 的代码，这 30 分钟就是「过早优化的成本」。在 Phase 5 复盘时，评估这笔投资是否值得。

### Q2：我不确定某个功能算不算技术债，怎么办？

**判断标准**：如果这个「债」让你每次加新功能都多花 10 分钟以上，就是技术债。如果只是「看着不舒服」但不影响开发效率，可以等 Phase 4 再处理。

### Q3：SDD 文档写得太复杂了，有必要吗？

**建议**：SDD 的核心价值不是文档本身，而是**系统性思考的过程**。如果你的项目很小（<1000 行），可以只写 L3（Plan）和 L4（Tasks）。如果项目很大，L1（宪法）能帮你避免架构级别的错误决策。

### Q4：我用 AI 生成的代码，算是我做的吗？

**答案**：算。AI 是工具，就像编译器一样。决定代码质量的是你的**决策**——选择什么技术栈、如何组织模块、何时重构、如何权衡。这些决策能力才是真正的工程能力。

### Q5：每个阶段的时间不够用，怎么办？

**建议**：时间建议是参考值。如果你 Phase 1 花了 40 分钟，可以从 Phase 4 借时间。关键是**完成整个周期**，不是每个阶段都完美。宁可 Phase 4 做得简单一点，也不要跳过某个阶段。

## 附录 D：技术债利率计算器

当你犹豫是否要重构时，用这个公式计算：

```
技术债日利息 = (混乱代码导致的额外分钟数 / 60) × 团队人数 × 时薪

例：
- 每次加功能多花 30 分钟
- 团队 3 人
- 时薪 100 元

日利息 = 0.5 小时 × 3 人 × 100 元 = 150 元/天

如果重构需要 8 小时（800 元成本）：
回本天数 = 800 / 150 ≈ 5.3 天

结论：如果你预期这个项目还要开发 6 天以上，现在重构是划算的。
```

**经验法则**：当你开始计算技术债时，通常已经到晚还的时候了。
